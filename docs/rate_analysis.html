Throughput: measurement & modeling

A deep-dive to replace ad-hoc heuristics with a defensible methodology. All data below is measured in the real game (headless 2.0.77, deterministic, fresh-world hardware, expansions off) via scripts/rate_study.py; every figure is reproducible from out/rate_study/*.json.

Proposed decisions (for review)
  1. Measurement: adopt MSER-5 truncation + batch-means confidence intervals (the standard simulation output-analysis method) with run-length control: extend the run until the CI half-width is < 3% of the mean. Retire the split-half/fixed-window heuristics.
  2. Primitive capacities: calibrate from micro-benchmarks measured in the game (cached per game version), not analytic formulas — the game quantizes inserter swings to whole ticks and belt-pickup differs from chest-pickup; formulas can't keep up.
  3. Allocation model: predict multi-consumer sharing with belt-position priority (upstream tap first), not proportional splits — measured contention is a 17:1 split, unambiguous.
  4. Prediction architecture: a three-layer stack — static bounds from dumps (design-time), a calibrated priority-flow model (fast metadata), and game-in-the-loop simulation (ground truth for anything that matters). The layers cross-check each other; no layer pretends to be what it isn't.

1 · What we measure, and why it went wrong

The harness samples every output chest once per game-second and fits a rate to the cumulative series. The first estimator was a fixed window ("last 60% of samples") — which silently assumes the series is steady inside that window. It isn't: warm-up on shared tapped belts spans minutes, because belts fill consumers in priority order (§3).

2 · The right estimator: MSER-5 + batch means

This is a textbook problem — steady-state output analysis of a discrete-event simulation. The standard treatment: delete the initial transient with MSER-5 (truncate where the marginal standard error of the retained mean is minimized — no thresholds to hand-tune), then compute a batch-means confidence interval on what remains (batching defeats the autocorrelation that makes naive OLS standard errors meaningless). Determinism makes repeated runs worthless as error bars; the CI measures within-run stability, and run-length control (extend until CI < 3% of mean) replaces guessed tick budgets.

Finding. On a long-enough run all sane estimators agree (0.150 ± tight CI) — the discipline isn't in the slope formula, it's in refusing to report before steady state is established and in saying how sure you are. MSER-5 gives both mechanically.

3 · Allocation is positional, not proportional

A purpose-built probe: one gear machine, its output belt tapped by two identical transport-belt assemblers. If sharing were proportional they'd split ~50/50.

Finding. The upstream tap runs at its own arm's full rate (0.857/s — exactly the pure-inserter swing rate, §4); the downstream tap gets the leftovers (0.050/s). Sharing on a tapped belt is priority-by-position. This also explains science_3's equilibrium: red science hit its solo machine cap while green idled for 8 minutes — red's taps sit upstream. Any prediction model using proportional splits is structurally wrong on contended lanes; conversely, priority allocation is deterministic and cheap to compute from the layout's tap order.

4 · Primitive capacities: measure, don't derive

Micro-benchmarks isolating one mechanism each, against the analytic formulas the rate model used:

primitiveanalyticmeasuredexplanation
inserter, chest→chest0.840/s0.857/s the game rounds each half-swing to whole ticks: 60/70 exactly
long-handed, chest→chest1.200/s1.204/s 60 ÷ 50 ticks ≈ 1.2 — quantization is kinder here
inserter, belt→chest(no model)0.938/s belt pickup shortens the swing: 60/64 ticks — faster than chest pickup
loader + belt path15.0/s15.03/s belt physics model is exact
Finding. Belt-and-loader math is exact; inserter timing is game-engine arcana (tick rounding, pickup geometry) that an analytic formula will chronically mispredict by 2–12%. The general solution is a calibration table measured by these micro-benchmarks, cached per game version — same philosophy as the FBSR dumps: real data over reimplementation.

5 · Proposed architecture

layersource of truthanswerscost
L1 static boundsprototype dumps machine caps, belt caps, Stage-B sizing (machine counts, lane counts, arms) instant
L2 priority-flow modelL1 + calibrated primitives + layout tap orderper-output steady rates & bottlenecks for metadata (blueprint tooltip, landing cards)ms
L3 game-in-the-loopheadless Factorio + MSER-5/batch-means ground truth: measured rates with CIs, warm-up times, regressions in CI ~2 s per game-minute

Each layer validates the one above it: L3 measured the 17:1 contention split that falsified L2's proportional assumption; the micro-benchmarks calibrate L2's constants; L2's bottleneck math sizes L1's Stage-B solver. Metadata should carry its provenance ("L2 estimate" vs "L3 measured ± CI").
